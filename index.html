Half-hourly electricity prices (Octopus Agile) joined with grid carbon intensity (National Grid ESO) for all 14 UK regions — normalised onto one score, run through a window optimiser, and served as plain JSON you can hit from anything. No keys. No SDK. One GET.

# Quickstart

# The best 3-hour window to charge in London (region C)
curl -s https://kennethmacfadyen-globemaker.github.io/wattwindow/api/v1/regions/C.json \
  | jq '.windows["3h"].best'

# Cheapest region to be in right now
curl -s https://kennethmacfadyen-globemaker.github.io/wattwindow/api/v1/best.json \
  | jq '.regions | min_by(.now.price) | {region, price: .now.price}'
// Should the dishwasher start now, or wait?
const BASE = "https://kennethmacfadyen-globemaker.github.io/wattwindow/api/v1";
const { current, windows } = await (await fetch(`${BASE}/regions/C.json`)).json();

const w = windows["2h"].best;
if (current.score >= 75) {
  console.log(`Run it now — score ${current.score}, ${current.price}p/kWh`);
} else {
  console.log(`Wait until ${w.from} (avg ${w.avgPrice}p/kWh, saves ${w.savingsVsAvgPct}%)`);
}
# Nightly EV charge: pick the cheapest 4-hour window
import requests

BASE = "https://kennethmacfadyen-globemaker.github.io/wattwindow/api/v1"
data = requests.get(f"{BASE}/regions/M.json", timeout=10).json()  # M = Yorkshire

w = data["windows"]["4h"]["cheapest"]
print(f"Charge {w['from']} → {w['to']}  ({w['avgPrice']}p/kWh, {w['avgCarbon']}g CO₂)")

# The maths, in the open

1 · Join

Every half-hour where both an Agile price and a regional carbon forecast exist is joined on its UTC boundary — typically 33–90 half-hours of horizon depending on when Octopus publishes tomorrow's prices (~16:00 UK).

2 · Score

score = 100 − (0.6·P + 0.4·C) where P and C are the price and carbon percentile ranks across the visible horizon. Percentiles, not raw values — so the score stays meaningful whether it's a windy bank holiday or a still January evening.

3 · Optimise

A sliding window over the series finds, for each duration (1–8h), the contiguous run with the best average score, lowest average price, and lowest average carbon — three answers, because “best” depends on whether you're paying or the planet is.

Weights favour price 60/40 because that's what moves behaviour — but both raw series ship in every response, so you can re-weight to taste. The pipeline source is public in the repo.

# Questions

Why trust the numbers?

Prices come from Octopus Energy's public Agile tariff feed (the same figures Agile customers are billed on); carbon comes from National Grid ESO's official forecast. WattWindow adds the join, the normalisation and the optimiser — the raw values pass through untouched, and both raw series are in every response so you can check.

What if I'm not on Agile?

The carbon signal and the shape of the price curve are still the best available proxy for when the grid is under- or over-supplied. Flat-tariff users cut carbon (and grid stress) by shifting; Agile users cut real money.

How fresh is it?

A pipeline regenerates every endpoint every 30 minutes (the meta.updated field tells you exactly when). Octopus publishes tomorrow's prices around 16:00 UK, so the horizon grows through the afternoon.

Uptime? SLA?

Endpoints are static files on GitHub's global CDN — there is no server to fall over. If GitHub Pages is up (it very reliably is), WattWindow is up. The commercial licence is honesty-priced accordingly.

Which region am I? (30-second answer)

Look at any electricity bill for the boxed “Supply Number” (MPAN) — the first two digits are your region, definitively:

10→A Eastern England · 11→B East Midlands · 12→C London · 13→D Merseyside & N. Wales · 14→E West Midlands · 15→F North East · 16→G North West · 17→P North Scotland · 18→N South Scotland · 19→J South East · 20→H Southern · 21→K South Wales · 22→L South West · 23→M Yorkshire

The mapping also ships in regions.json as mpanPrefix, so your app can do this for your users. (Postcodes don't map cleanly to DNO boundaries — the MPAN is the ground truth.)
